## What is the Legacy Migration Task Manager?

The Legacy Migration Task Manager is a comprehensive school project that showcases the practical transformation of a legacy task management system from a tightly-coupled monolithic architecture to a modern, maintainable MVC (Model-View-Controller) architecture.

This project demonstrates real-world architectural patterns and best practices for separating concerns, improving code organization, and building scalable web applications with a clear separation between backend API and frontend interface.

<Note>
This is an educational project designed to demonstrate architectural migration concepts and modern web development practices.
</Note>

## Why this project exists

The primary goal of this project is educational:

1. **Demonstrate architectural migration**: Show how to transform a monolithic application into a modern MVC architecture
2. **Apply separation of concerns**: Implement clear boundaries between presentation, business logic, and data layers
3. **Practice modern development**: Use contemporary tools and frameworks (.NET 10, React, PostgreSQL)
4. **Improve maintainability**: Create a codebase that's easier to test, modify, and scale

<Info>
The project evolved from a single, tightly-coupled codebase to a clean separation between:
- Backend API (C# .NET 10)
- Frontend interface (React.js)
- Database layer (PostgreSQL)
</Info>

- **C# with .NET 10**: Modern framework providing robust API capabilities with native OpenAPI support

- **Railway**: Cloud platform for deployment and hosting
- **Git**: Version control and collaboration

## Architecture highlights

The application follows the MVC pattern with additional service layer:

```
┌─────────────────┐
│   React View    │  ← User Interface (Components)
└────────┬────────┘
         │
         ↓
┌─────────────────┐
│  Controllers    │  ← HTTP Request Handlers
└────────┬────────┘
         │
         ↓
┌─────────────────┐
│    Services     │  ← Business Logic Layer
└────────┬────────┘
         │
         ↓
┌─────────────────┐
│  Models/DTOs    │  ← Data Layer
└────────┬────────┘
         │
         ↓
┌─────────────────┐
│   PostgreSQL    │  ← Data Persistence
└─────────────────┘
```

This separation enables:

- **Independent development**: Frontend and backend teams can work in parallel
- **Easy testing**: Each layer can be tested in isolation
- **Scalability**: Components can be scaled independently based on load
- **Maintainability**: Changes in one layer don't cascade to others
